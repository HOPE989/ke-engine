## Context

The current ingestion pipeline accepts supported documents, converts them to Markdown, and lets callers manually chunk converted documents. Chunking persists `knowledge_segment` rows and advances the document to `CHUNKED`, but it deliberately does not create embeddings or write Elasticsearch vectors.

The next stage must convert embeddable segments into Elasticsearch vector documents. The existing backend already has the right local boundaries for this: document-owned repository/workflow modules, Kafka dispatch and worker infrastructure, Redis document locks, OpenAI-compatible LangChain usage, and Docker-provided Elasticsearch. This design follows the Java reference implementation's embed-and-store semantics where `embedding_id` stores the ID returned by the vector store after insertion.

## Goals / Non-Goals

**Goals:**

- Dispatch vector storage after successful document chunking as the normal application path.
- Add a manual API trigger for vector storage of one `CHUNKED` document for end-to-end testing and operational retries.
- Dispatch vector-storage work through Kafka instead of performing it in the request.
- Use a dedicated Kafka worker and Redis per-document lock to process one document at a time.
- Generate embeddings with `OpenAIEmbeddings(model="text-embedding-v4")`.
- Use `OpenAIEmbeddings.chunk_size = 9` and DB page size `100`.
- Store Elasticsearch-generated vector document IDs in `knowledge_segment.embedding_id`.
- Treat `knowledge_segment.text` as vector document page content and `knowledge_segment.metadata` as vector metadata.
- Advance non-skipped segments from `STORED` to `VECTOR_STORED`.
- Advance the document from `CHUNKED` to `VECTOR_STORED` only after a final double-check confirms no remaining embeddable `STORED` segments.
- Preserve all database status updates for one document in a single long database transaction and use Elasticsearch cleanup compensation for external side effects.

**Non-Goals:**

- No retrieval, ranking, or chat-answer integration.
- No scheduled compensation job in the first implementation.
- No dead-letter queue for malformed or poison messages in the first implementation.
- No document versioning or version switch behavior.
- No deterministic Elasticsearch IDs; this change uses IDs returned by Elasticsearch/LangChain.
- No strict distributed transaction across PostgreSQL, Elasticsearch, OpenAI, and Kafka.

## Decisions

### 1. Trigger vector storage through Kafka

Successful chunking should publish `document.embed_store.requested` after the chunking database transaction commits. This is the normal application path.

The manual API endpoint remains useful for end-to-end testing and explicit retry. It validates the document and publishes the same Kafka event, then returns after dispatch; it does not call the embedding model or Elasticsearch directly.

This mirrors the existing conversion worker pattern and avoids long-running embedding work inside the HTTP request or chunking request.

### 2. Use `STORED` as the segment post-chunking status

`STORED` means the segment is stored in the relational database. Chunking should create all segment rows with `status = STORED`; embeddable child or normal segments later move to `VECTOR_STORED`. Parent segments with `skip_embedding = true` stay `STORED`.

The document remains `CHUNKED` after chunking and advances to `VECTOR_STORED` after embed-and-store completion.

### 3. Keep Elasticsearch IDs generated by the vector store

The Java reference implementation calls `embeddingStore.addAll(...)` and stores the returned IDs on each segment. The Python implementation should follow the same semantic contract: `embedding_id` is the vector-store document ID returned after insertion, not the embedding vector content and not necessarily `chunk_id`.

This means retry after partial failure can create orphaned vectors. The design mitigates that with cleanup before processing and cleanup on failure.

### 4. Separate DB page size from embedding API chunk size

The worker should scan up to 100 database segments per loop. It should configure `OpenAIEmbeddings.chunk_size = 9`, letting LangChain split the 100 texts into embedding API requests of at most 9 items and combine the results in input order.

`chunk_size = 9` supersedes the initial draft's suggested value and is the final decision from requirement discussion. It matches the Java reference's `maxSegmentsPerBatch = 9` and keeps provider batch size conservative for the OpenAI-compatible embedding endpoint. `chunk_size = 9` and DB page size `100` are module constants. `embedding_dimensions = 1536` is configurable because it must match both the embedding provider and Elasticsearch dense-vector mapping.

### 5. Query the first page repeatedly

Each loop should select the first 100 rows matching:

```text
document_id = doc_id
status = STORED
skip_embedding = false
embedding_id IS NULL
ORDER BY chunk_order ASC, id ASC
LIMIT 100
```

The worker must not use offset pagination because each successful batch changes the status of selected rows. Re-querying the first page of the remaining set avoids skipping rows.

### 6. Use one long DB transaction for document-level database completion

The user-facing requirement is that the document vector-storage task succeeds or fails as a whole from the database perspective. The worker should hold one database transaction across all segment updates and the final document status update. If any step fails, the transaction rolls back and the document remains `CHUNKED` with segments still eligible for retry.

OpenAI and Elasticsearch calls happen while that transaction is open. This trades transaction duration for simpler all-or-nothing database semantics.

### 7. Compensate Elasticsearch because this is not a distributed transaction

PostgreSQL cannot roll back vectors already written to Elasticsearch. The worker should:

- delete existing Elasticsearch vectors for `metadata.docId = doc_id` before processing a `CHUNKED` document;
- track vector IDs returned during the current attempt;
- on failure, try to delete returned vector IDs and then fall back to deleting by `metadata.docId`;
- not commit the Kafka message when the task fails.

This gives eventual consistency across retries, while the database remains all-or-nothing for the document.

### 7a. Commit only terminal worker outcomes

The worker should commit Kafka messages for terminal, non-retryable outcomes:

- vector storage completes successfully;
- the document no longer exists;
- the document is already `VECTOR_STORED`;
- the document is in a business state other than `CHUNKED` or `VECTOR_STORED`.

The worker should leave messages uncommitted for retryable outcomes:

- the document vector-storage lock is busy;
- OpenAI, Elasticsearch, Redis, database, or other infrastructure calls fail;
- Elasticsearch returns a vector ID count that does not match the processed segment count;
- final double-check fails.

Kafka payloads are produced internally by typed dispatchers in this change. Malformed payload handling is intentionally not part of this scope.

### 8. Final double-check gates document completion

After the processing loop, the worker must count remaining embeddable segments:

```text
document_id = doc_id
status = STORED
skip_embedding = false
embedding_id IS NULL
```

Only a zero count allows the document to advance to `VECTOR_STORED`. A nonzero count is a task failure and must roll back the database transaction.

### 9. Store text and metadata separately in Elasticsearch

The vector document should store:

```text
page_content/text = knowledge_segment.text
metadata = knowledge_segment.metadata
```

Segment text must not be duplicated into metadata. Metadata must include enough fields for later filtering and cleanup, especially `docId`, `chunkId`, and `accessibleBy`.

## Risks / Trade-offs

- Long database transaction during OpenAI and Elasticsearch calls -> Accept for this change because the task requires all-or-nothing database semantics. Use a Redis document lock to prevent concurrent work on the same document.
- Elasticsearch writes can survive DB rollback -> Clean by returned vector IDs and by `metadata.docId`; also clean by `metadata.docId` before each retry.
- Worker crash after ES writes but before cleanup -> The next retry for the still-`CHUNKED` document cleans by `metadata.docId` before writing new vectors.
- Using generated Elasticsearch IDs reduces idempotence -> Kept intentionally to match the Java reference implementation and because `embedding_id` stores vector-store returned IDs.
- Holding a Kafka message until full document completion can cause repeated retries -> Required so failed documents remain retryable.
- Missing or stale Elasticsearch index mapping can break vector writes -> Add configuration and startup/test coverage for `elasticsearch_url`, `elasticsearch_index`, and `embedding_dimensions`.

## Migration Plan

1. Add a migration that extends `knowledge_document.status` to include `VECTOR_STORED` and changes `knowledge_segment.status` default from `INIT` to `STORED`.
2. Update ORM models and tests to reflect `DocumentStatus.VECTOR_STORED` and segment `STORED`/`VECTOR_STORED`.
3. Existing local test databases can be reset through the current development database initialization flow. If preserving existing data, migrate existing `knowledge_segment.status = INIT` rows for `CHUNKED` documents to `STORED`.
4. Add Elasticsearch/vector-store settings and dependency installation before running the worker.

## Open Questions

- The first implementation will not create a scheduled compensation job. If operational usage needs it, add a later change that scans `CHUNKED` documents and re-dispatches vector-storage events.
