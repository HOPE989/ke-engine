# Async Document Upload Design

## Goal

Decouple document upload from document processing. The upload request should return
quickly after validation, persistence, original-file storage, and task scheduling.
MinerU parsing, ZIP extraction, converted asset upload, and future ingestion stages
should run in background workers.

This design also changes `doc_id` from a database identity value to an
application-generated Snowflake `BIGINT`. The column keeps the name `doc_id`.

## Scope

In scope:

- Replace database-generated `knowledge_document.doc_id` with Snowflake-generated
  `BIGINT` IDs.
- Keep `doc_id` as the public document identifier in API responses, MinIO object
  keys, Celery task arguments, and status lookup routes.
- Make `POST /api/v1/document/upload` uniformly asynchronous.
- Return `202 Accepted` once the original object is stored and processing is
  scheduled or durably recoverable.
- Add background document processing with Celery and Redis.
- Add recoverability for MQ publish failures and stuck processing records.
- Preserve the existing MinIO object layout under `documents/{doc_id}/`.
- Preserve current PDF MinerU conversion behavior inside the worker.

Out of scope:

- Authentication and authorization enforcement.
- Chunking, embedding, vector storage, and document search.
- User-facing realtime push notifications.
- Dynamic Snowflake worker-id allocation through Redis or Postgres.
- Splitting `doc_id` into separate internal and public identifiers.

## Key Decisions

### `doc_id` remains the public primary identifier

`knowledge_document.doc_id` remains the primary key and the API-visible ID, but it
is no longer generated by Postgres identity/sequence behavior.

The upload boundary generates `doc_id` before inserting the `INIT` row:

```text
doc_id = snowflake.next_id()
INSERT knowledge_document(doc_id, ...)
```

This keeps the rest of the system simple:

- API response uses `doc_id`.
- MinIO object keys use `documents/{doc_id}/...`.
- Celery tasks carry only `doc_id`.
- Status lookup routes use `doc_id`.

### Snowflake is configured explicitly

The first version uses environment configuration rather than dynamic worker-id
assignment:

```text
SNOWFLAKE_WORKER_ID=1
SNOWFLAKE_EPOCH_MS=1767225600000
```

Use this bit layout:

```text
1 bit   sign bit, always 0
41 bit  milliseconds since project epoch
10 bit  worker id, 0..1023
12 bit  sequence per millisecond, 0..4095
```

Every process that generates `doc_id` must have a unique `SNOWFLAKE_WORKER_ID`.
For this change, the FastAPI API process is the only required generator. Workers
process existing IDs and do not need to generate document IDs unless later
features add worker-created records.

### Upload is uniformly asynchronous

`POST /api/v1/document/upload` should not synchronously wait for text conversion
or PDF conversion. Even plain text files go through the same queued processing
model to keep client behavior consistent.

Success response:

```text
HTTP 202 Accepted
```

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "doc_id": 739482091234567168,
    "doc_title": "guide.pdf",
    "upload_user": "alice",
    "accessible_by": "team-a",
    "doc_url": "http://localhost:9000/documents/documents/739482091234567168/original/guide.pdf",
    "converted_doc_url": null,
    "status": "QUEUED"
  }
}
```

The exact success envelope can continue using the shared `APIResponse`; the
important change is the HTTP status and the `status` value.

## State Model

The current states are too narrow for background processing. The async flow needs
explicit terminal and retryable states:

```text
INIT
  -> UPLOADED
  -> QUEUED
  -> PROCESSING
  -> CONVERTED

PROCESSING
  -> FAILED

QUEUED / PROCESSING
  -> RETRY_PENDING
  -> QUEUED
```

Replace the current `CONVERTING` status with `PROCESSING`. Format-specific
details belong in the stored document type, job metadata, and logs rather than in
separate document lifecycle states.

First-version statuses:

- `INIT`: database row exists; original object is not confirmed.
- `UPLOADED`: original object is stored.
- `QUEUED`: background processing has been scheduled or marked for recovery.
- `PROCESSING`: a worker has claimed the document.
- `CONVERTED`: converted document URL is published.
- `FAILED`: background processing reached a terminal failure after retry policy.

Expected-state updates remain the concurrency guard. Workers should transition:

```text
QUEUED -> PROCESSING -> CONVERTED
QUEUED -> PROCESSING -> FAILED
```

If a worker cannot claim `QUEUED -> PROCESSING`, it exits without processing.

## Data Flow

```text
POST /api/v1/document/upload
  -> validate multipart fields and file bytes
  -> detect document file type
  -> generate Snowflake doc_id
  -> insert knowledge_document(status=INIT, file_type=detected type)
  -> upload original bytes to MinIO
  -> mark UPLOADED with doc_url
  -> mark QUEUED
  -> create processing job marker in PENDING
  -> enqueue Celery process_document(doc_id)
  -> if publish succeeds, mark job ENQUEUED
  -> if publish fails, leave job PENDING for recovery
  -> return 202 with doc_id and status

Celery process_document(doc_id)
  -> load knowledge_document by doc_id
  -> claim QUEUED -> PROCESSING
  -> process by stored file_type
  -> upload converted objects
  -> mark CONVERTED with converted_doc_url
  -> on retryable failure, retry through Celery policy
  -> on terminal failure, mark FAILED
```

The worker reads the original file from object storage instead of receiving file
bytes through MQ. Celery messages stay small and retry-safe.

## Durable Scheduling

Publishing to Redis is not transactional with the database. The design should not
leave a document permanently invisible if this sequence fails:

```text
mark UPLOADED
publish Celery task
```

Use a durable job marker table. Add `document_processing_job` with `doc_id` as a
one-to-one processing record for each document:

```text
doc_id BIGINT primary key references knowledge_document(doc_id)
status PENDING | ENQUEUED | RUNNING | SUCCEEDED | FAILED
attempt_count integer not null default 0
last_error varchar/text null
next_run_at timestamp null
created_at timestamp not null
updated_at timestamp not null
```

Upload marks the document `QUEUED`, then creates the job marker in `PENDING`
before publishing. If Celery publish succeeds, mark the job `ENQUEUED`. If
publish fails, leave the job `PENDING`; periodic recovery later enqueues it.

Upload error behavior:

- If original upload fails: return `502 document storage failed`, document stays
  `INIT`.
- If job marker creation fails: return `500 document persistence failed`.
- If Celery publish fails after durable `PENDING`: mark document `QUEUED` and
  return `202`; periodic recovery is responsible for publishing.

This makes `202` mean "accepted by durable ingestion," not "already present in
Redis."

## Periodic Recovery

Celery Beat or another scheduler should run a small recovery task:

```text
recover_document_processing_jobs
  -> find PENDING jobs due for enqueue
  -> publish process_document(doc_id)
  -> mark ENQUEUED
  -> find stale RUNNING/PROCESSING jobs past timeout
  -> move retryable jobs back to PENDING/QUEUED or mark FAILED
```

The scanner must use database expected-state updates or row locks so multiple
schedulers do not enqueue the same job repeatedly. Duplicate Celery messages are
still acceptable because workers claim work with `QUEUED -> PROCESSING`.

## API Contract

Add a status lookup endpoint before relying on async upload in clients:

```text
GET /api/v1/document/{doc_id}
```

Response data should reuse `DocumentMetadata`:

```json
{
  "doc_id": 739482091234567168,
  "doc_title": "guide.pdf",
  "upload_user": "alice",
  "accessible_by": "team-a",
  "doc_url": "...",
  "converted_doc_url": null,
  "status": "PROCESSING"
}
```

`POST /upload` returns the same shape with `status=QUEUED`. Clients poll the
lookup route until `CONVERTED` or `FAILED`.

## Error Handling

Request-bound validation errors keep the existing response messages:

- `invalid upload request`
- `file too large`
- `unsupported file type`
- `file type detection failed`
- `document persistence failed`
- `document storage failed`

Background processing failures are not returned from `POST /upload`. They are
persisted on the document/job record and surfaced through status lookup:

```text
status = FAILED
converted_doc_url = null
```

The first version can keep failure details internal and expose only a stable
document status. `document_processing_job.last_error` is internal and sanitized;
the public status API does not expose provider response bodies, local paths, or
stack traces.

## Implementation Boundaries

Suggested module ownership:

```text
backend/app/infrastructure/snowflake.py
  SnowflakeIdGenerator

backend/app/infrastructure/celery.py
  Celery app construction and config

backend/app/modules/document/tasks.py
  Celery tasks for document processing and recovery

backend/app/modules/document/workflow.py
  request-bound upload acceptance workflow

backend/app/modules/document/processing.py
  worker-side processing workflow

backend/app/modules/document/repository.py
  expected-state updates and job marker persistence
```

MinerU client code can remain in `app.infrastructure.mineru`. The blocking
polling behavior moves out of the HTTP request path because it runs inside the
Celery worker.

## Migration Notes

Because the feature is not live, use a direct migration adjustment rather than a
backward-compatible dual-ID migration:

- Change `knowledge_document.doc_id` from `BIGINT IDENTITY` to plain `BIGINT`.
- Keep it as the primary key.
- Add a stored `file_type` column so workers do not re-run Magika.
- Update tests that inspect the migration to expect no identity behavior.
- Update repository creation to require a generated `doc_id`.
- Update any tests that assumed database-generated IDs.
- Do not add a manual retry endpoint in this change; retry management is a later
  API capability.

If existing local development data matters, reset the dev database or provide a
one-off migration that fills missing IDs before dropping identity behavior.

## Testing

Focused tests should cover:

- Snowflake ID monotonicity within one process.
- Snowflake worker-id validation and sequence rollover behavior.
- Migration metadata: `doc_id` is `BIGINT` primary key without identity.
- Upload acceptance returns `202` and `QUEUED`.
- Upload generates `doc_id` before persistence and object-key construction.
- Plain text upload is queued rather than synchronously converted.
- PDF upload does not call MinerU from the request path.
- Celery task processes plain text to `CONVERTED`.
- Celery task processes PDF through MinerU to `CONVERTED`.
- Worker duplicate-message handling via expected-state claim.
- MQ publish failure with durable job marker is recoverable.
- Periodic recovery re-enqueues pending jobs and handles stale processing jobs.
