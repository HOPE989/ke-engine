# 会话标题异步生成设计

**日期：** 2026-07-16
**状态：** 已确认，待实现

## 目标

用户发送新会话的首条消息时，立即用规范化消息的前 20 个字符创建可展示的标题，并在数据库事务提交后异步调用固定的 `qwen3.6-flash` 模型生成正式标题。标题任务不得阻塞或影响正常的流式回答。

## 范围

本次只实现新会话的一次性自动标题生成：

- 新会话创建时写入前 20 个字符作为临时标题。
- 事务提交后直接创建进程内异步任务。
- 正式标题最多 20 个字符。
- 已有会话追加消息时保留原标题，不触发标题更新。
- 标题失败时保留临时标题。

本次不实现：

- 多轮对话自动重生成标题；
- 用户手工修改标题；
- 标题生成重试或持久化任务队列；
- 标题完成事件、前端轮询或实时推送；
- 新数据库字段或 Alembic 迁移。

## 核心决策

### 标题任务不区分首轮与多轮

标题生成函数只接收会话标识和标题上下文，不包含 `is_first_turn` 等状态。当前是否提交任务由 `ConversationService.accept_user_turn()` 的创建会话分支决定；未来如果支持多轮更新，可以复用同一生成函数并传入多轮上下文。

`AcceptedUserTurn` 不增加标题相关字段。

### 临时标题只属于新会话创建

`conversations.title` 当前不可为空。新会话创建时，`ConversationService` 使用规范化后首条消息的 `content[:20]` 作为临时标题。已有会话已经有标题，追加消息时不得覆盖或重新生成临时标题。

数据库列继续保持 `VARCHAR(255)`，应用层只将临时标题和正式标题限制为最多 20 个字符。

### 事务提交后直接创建异步任务

标题任务使用 `asyncio.create_task()` 提交到当前 FastAPI/Uvicorn worker 正在运行的事件循环。它不是新线程、新进程或独立事件循环。

任务不能在 USER 消息事务提交前创建。`accept_user_turn()` 在事务内记录待提交的标题请求，退出事务上下文并确认提交成功后才调用标题任务提交函数。由此保证标题任务引用的会话和 USER 消息已经存在。

标题任务使用独立的 `AsyncSession`，不得复用请求事务的 Session。

### 不引入专用调度器

标题属于低重要性、尽力而为的派生数据，临时标题已经提供降级结果。因此不新增 `TitleScheduler`，不跟踪业务状态，不在应用关闭时专门等待，也不实现超时和重试。

提交函数只做三件事：

1. 使用 `asyncio.create_task()` 创建任务；
2. 将任务加入模块级集合，保持执行期间的强引用；
3. 任务完成后通过 `add_done_callback(set.discard)` 释放强引用，让 Python 后续回收 task 对象。

标题协程内部捕获并记录普通异常，避免出现未读取的 task 异常。`CancelledError` 保持取消语义并继续抛出。

## 数据流

1. Router 调用 `ConversationService.accept_user_turn()`。
2. Service 规范化用户内容。
3. 当 `conversation_id is None` 时，创建会话并写入 `content[:20]` 临时标题。
4. Service 在同一事务中写入 USER 消息。
5. 事务成功提交后，Service 直接提交标题异步任务。
6. Service 返回现有 `AcceptedUserTurn`，Router 按现有流程启动 `CompletionProducer`。
7. 标题任务和 completion 在同一 worker 的事件循环上并发推进。
8. 标题任务异步调用 `qwen3.6-flash`。
9. 输出清洗成功后，标题任务打开新的短数据库事务更新 `conversations.title`。

模型调用期间不得持有数据库连接或数据库事务。

## 组件变更

### ConversationService

- 将 `_title_from_content()` 的截取上限从 255 改为 20。
- 构造新会话时记录标题生成请求。
- 在事务提交后调用标题任务提交函数。
- 已有会话分支不修改标题，也不提交标题任务。

### 标题生成模块

新增独立标题模块，包含：

- 固定模型名 `qwen3.6-flash`；
- 标题提示词；
- 标题请求数据结构；
- 输出清洗函数；
- 异步模型调用和数据库更新函数；
- 直接创建 task 的提交函数和后台 task 强引用集合。

当前标题上下文直接使用 `accept_user_turn()` 已经规范化的首条 USER 消息，避免额外查询数据库，也避免异步读取时混入随后生成的 ASSISTANT 消息。

### Chat API 依赖装配

Chat API lifespan 使用现有 OpenAI-compatible API Key 和 Base URL，再创建一个模型名固定为 `qwen3.6-flash` 的 Chat model。标题模型放入 `ChatApiDeps`，供 `ConversationService` 提交任务时使用。

标题模型客户端的创建不执行标题请求。应用关闭时不为标题 task 增加专用等待流程。

### ConversationRepository

增加内部标题更新方法。更新应满足：

- 只匹配指定且未删除的会话；
- 只改变 `title`；
- 明确保留原 `updated_at`，避免标题完成时间改变会话列表的活跃排序；
- 会话不存在时安静结束，不影响主请求。

## 模型提示与输出清洗

系统提示要求模型根据用户消息概括会话主题，只返回标题，不解释、不添加引号，并限制在 20 个字符以内。

服务端不信任模型完全遵守格式，依次执行：

1. 确认响应内容可以转换为非空文本；
2. 去除首尾空白；
3. 只取第一行；
4. 去除常见的 `标题：` 或 `标题:` 前缀；
5. 去除成对的外层中英文引号；
6. 再次去除首尾空白；
7. 截断到 20 个字符。

清洗结果为空时不更新数据库，继续保留临时标题。

## 错误与可靠性边界

标题任务与 completion 相互独立：标题失败不使 completion 返回错误，completion 失败也不取消标题任务。客户端断开 SSE 后，只要 worker 仍在运行，标题 task 可以继续执行。

标题任务是进程内 best-effort 任务：

- 模型或数据库失败时记录 `conversation_id` 和异常，不记录完整用户内容；
- 不自动重试；
- worker 崩溃、强制退出或事件循环关闭时，未完成任务允许丢失；
- 任务丢失后临时标题继续作为最终降级标题；
- 多 worker 部署时，任务只存在于接收请求的 worker 中。

## 前端可见性

第一版不增加标题更新通知。前端继续展示数据库当前标题：新会话先显示临时标题，正式标题写入后在下一次会话列表刷新时可见。如果 completion 结束后的现有列表刷新早于标题任务完成，用户会暂时继续看到临时标题，直到后续刷新。

## 测试

- 新会话临时标题使用规范化内容的前 20 个字符。
- 已有会话追加消息时原标题不变且不提交标题任务。
- 标题任务只在事务成功提交后创建。
- USER 消息事务回滚时不创建标题任务。
- 标题模型名称固定为 `qwen3.6-flash`。
- 标题输出的空白、首行、前缀、引号和 20 字截断规则正确。
- 空模型输出不覆盖临时标题。
- 模型异常和数据库异常不传播到 completion。
- 标题数据库更新不改变 `updated_at`。
- 后台 task 在运行期间有强引用，完成后从集合移除。
- 现有 conversation、completion 和 SSE 测试保持通过。

## 后续多轮扩展

未来支持多轮标题时，已有标题继续展示，新任务完成后再覆盖。标题生成函数仍可复用，但调用方需要提供整理后的多轮上下文，并增加基于消息版本或生成序号的条件更新，防止旧任务后完成而覆盖新标题。这些并发控制不在本次一次性生成范围内。
