# 项目中意图识别提示词的优化

## 1. 项目背景

ke-engine 是一个面向企业内部员工的业务知识引擎，主要服务于以下场景：

- 铁路运输和煤炭运输；
- 运行计划、装车、编组、发运和到达等生产调度业务；
- 煤炭采购、销售、合同和数质量结算；
- 企业制度、政策法规、调度规程和行业知识；
- 货单、运单、货票及经营指标等结构化业务数据。

系统需要先理解用户的问题，再选择不同的专业回答 Prompt，并为后续的 RAG 检索和结构化数据查询提取必要实体。

意图识别链路沿用了 know-engine 已经验证过的基本方式：

~~~text
会话历史 + 当前问题
        ↓
意图识别
        ↓
route / intent / entities / reasoning / clarification_question
        ↓
根据 intent 选择不同的专业回答 Prompt
~~~

这里没有额外引入 BusinessDomain。知识库中的“铁路、煤炭、港口、航运、电力、化工”等分类主要用于文档管理和检索过滤；意图分类则要回答另一个问题：

> 用户希望系统采用哪一种业务回答方式？

## 2. 初版意图识别 Prompt

项目早期为了快速验证链路，编写了一个比较朴素的意图识别 Prompt。

> 说明：这里的初版 Prompt 是依据 know-engine 原始模板整理的场景化迁移基线，用于说明优化起点和开展离线对照，不表示它已经作为 ke-engine 的生产版本上线。

~~~text
# 角色定义

你是一名企业知识库意图识别助手。请分析用户输入，判断问题是否与铁路运输、
煤炭运输、煤炭销售及企业知识相关，并识别用户意图和关键信息。

# 相关性判断

首先判断用户输入是否与企业业务相关。

如果不相关，输出：
{"related": false}

如果相关，继续识别意图。

# 意图分类体系

1. 铁路运输
   - 运行计划、装车、编组、发运、到站、铁路线路等问题。

2. 煤炭业务
   - 煤种、煤质、库存、煤炭运输等问题。

3. 销售业务
   - 客户、供应商、合同、订单、价格、结算等问题。

4. 企业知识
   - 政策法规、企业制度、调度规程、行业规章和业务知识。

5. 数据查询
   - 查询业务数据、统计数据和经营指标。

6. 闲聊与通用问答
   - 与企业业务无关的问题，此类 related 为 false。

7. 其他
   - 与企业业务有关，但无法归入以上类别的问题。

# 关键信息提取

提取以下信息，未提及则填写 null：

- order_id：订单、合同或业务单据编号；
- customer：客户名称；
- coal_type：煤种；
- station：车站；
- time：时间；
- description：问题描述。

# 输出格式

{
  "related": true,
  "intent": "用户意图",
  "entities": {
    "order_id": null,
    "customer": null,
    "coal_type": null,
    "station": null,
    "time": null,
    "description": null
  }
}
~~~

这个版本能够完成最基本的分类，但随着场景样例增多，开始暴露出明显问题。

## 3. 初版 Prompt 暴露的问题

### 3.1 按产业名词分类，无法表达用户真正想做什么

初版意图中的“铁路运输”“煤炭业务”“销售业务”更像业务领域，而不是用户意图。

例如下面四个问题都包含“运单”，但下游处理方式完全不同：

| 用户问题 | 真正诉求 |
|---|---|
| 运单应该包含哪些信息？ | 询问业务规则 |
| 查询运单 YD2026001 | 查询具体业务数据 |
| 统计本月各客户运单量 | 经营分析 |
| 这张运单为什么一直没有到站？ | 查询状态并分析异常 |

如果仅根据“运单”关键词分类为“铁路运输”，下游无法判断应该选择知识问答 Prompt，还是数据查询 Prompt。

### 3.2 业务范围描述过宽，容易产生相关性误判

初版 Prompt 只说明“与铁路运输相关”，没有明确企业业务边界。

因此：

~~~text
火车票怎么退？
~~~

可能因为包含“火车票”而被判断为铁路业务相关。但 ke-engine 服务的是企业煤炭运输和生产经营场景，不负责旅客购票、退票和出行服务。

相反：

~~~text
按实际版呢？
~~~

如果脱离上一轮“查询昨天装车计划的模拟版”这一上下文，又可能被判断为无意义输入，实际上它是一个明确的多轮业务查询。

### 3.3 企业知识目录被直接当成意图

知识库中包含政策法规、调度规程、煤炭购销、产业协调、专业知识等目录。

初版 Prompt 将它们统一归为“企业知识”，但以下问题需要不同的回答约束：

~~~text
《铁路货物运输规程》对货票填写有什么要求？
~~~

回答时必须优先引用正式制度和规程，避免脱离条文自由发挥。

~~~text
重车周转时间是什么意思？
~~~

回答重点是概念解释、计算口径和业务示例。

两者虽然都需要 RAG，但适用的 Prompt、证据约束和表达方式并不相同。

### 3.4 知识问答与数据查询边界不清

初版 Prompt 主要依赖“查询、多少、统计”等显式关键词，容易遗漏隐含的数据查询。

例如：

~~~text
今天神东站装了几列？
~~~

这是结构化数据查询。

~~~text
装车计划一般是怎么编制的？
~~~

这是运输生产知识问答。

两个问题都包含“装车”，但判断依据应该是用户在询问“规则与方法”，还是“特定时间和对象的事实数据”。

### 3.5 铁路业务实体被压缩成通用 order_id

铁路运输领域并不存在一个可以覆盖所有业务对象的通用“订单号”。实际业务中至少包括：

- 运行计划编号；
- 列车车次；
- 编组编号；
- 合同编号；
- 货单编号；
- 运单编号；
- 货票编号。

如果全部写入 order_id，下游无法确定应该查询哪张表、使用哪个字段，也无法准确生成澄清问题。

### 3.6 缺少数据版本和时间口径

ADS 数据中存在历史、实际、模拟等不同数据版本，并且大量指标依赖计划日期、统计周期和时间范围。

例如：

~~~text
查询昨天大柳塔站的装车计划。
~~~

如果没有提取“昨天”和数据版本，系统可能返回错误口径的数据。

多轮场景中的：

~~~text
按实际版呢？
~~~

更要求意图识别同时参考会话历史并提取 data_version。

### 3.7 缺乏易混淆场景辨析

初版 Prompt 只罗列每个类别覆盖的关键词，没有说明冲突时应该如何选择。

主要混淆包括：

- 政策制度与专业知识；
- 运输生产知识与具体数据查询；
- 铁路运输与煤炭购销；
- 企业铁路业务与公众出行问题；
- 当前完整问题与多轮省略问题；
- 货单、运单、货票与通用“订单”。

### 3.8 输出协议不够稳定

初版 Prompt 对输出结构约束较弱：

- 非业务问题只返回 related，业务问题返回完整对象，结构不统一；
- 没有 reasoning，难以分析误判原因；
- 没有限制模型创造新标签；
- 没有说明只能输出 JSON；
- 没有定义实体字段的具体含义；
- 缺少输入输出配对的 Few-shot。

这些问题会增加解析失败和评测困难。

上述问题主要来自三类材料的交叉检查：知识库目录反映的文档范围、ADS 表反映的结构化查询对象，以及运行计划、编组、货单、运单、货票等铁路业务表达。它们共同说明，仅替换汽车领域名词不能完成可靠迁移。

~~~json
  "route": "CLARIFY",
~~~

## 5. 优化思路

针对以上问题，Prompt 优化遵循以下原则。

### 5.1 从“属于哪个板块”改为“需要哪种回答方式”

意图不再直接使用“铁路、煤炭、销售”等产业名词，而是按照下游 Prompt 和执行方式划分：

1. POLICY_RULE_QA：政策制度与规程；
2. TRANSPORT_OPERATION_QA：运输生产与调度；
3. COAL_SALES_QA：煤炭购销与销售；
4. PROFESSIONAL_KNOWLEDGE_QA：专业知识与技术指导；
5. BUSINESS_DATA_QUERY：经营数据查询与分析；
6. OTHER_BUSINESS：其他业务。

这仍然是 know-engine 风格的扁平业务意图体系，没有引入独立 BusinessDomain。非业务和澄清不再伪装成业务 intent，而是分别由 NON_BUSINESS 和 CLARIFY route 表达。

### 5.2 明确企业业务边界

相关性判断不再依赖“铁路、煤炭”等关键词，而是要求问题与企业生产、运输、购销、制度或经营数据有关。

公众客运、股票投资、日常生活等问题，即使出现“火车”“煤炭”等词，也不属于本系统的业务范围。

### 5.3 增加固定的判定步骤

在输出最终结果前，要求模型依次完成：

1. 结合会话历史还原当前问题；
2. 判断是否属于企业业务范围；
3. 判断用户在询问规则、流程、专业原理还是具体数据；
4. 识别运输、购销和经营分析边界；
5. 提取铁路业务实体；
6. 输出固定 JSON。

reasoning 只保留简短、可审计的判定依据，不要求输出冗长思维链。

### 5.4 用真实业务对象替代通用 order_id

将运行计划、车次、编组、合同和运输单据分别表达。

货单、运单和货票采用：

~~~text
document_type + document_no
~~~

既能避免字段无限膨胀，也能让下游知道应该查询哪一类单据。

### 5.5 增加易混淆边界和正反例

Prompt 中重点解释：

- 规则依据与专业原理的区别；
- 业务流程与具体数据的区别；
- 运输执行与购销结算的区别；
- 企业货运与公众客运的区别；
- 单轮完整问题与多轮省略问题的处理方式。

~~~text

你是一名企业铁路运输与煤炭经营领域的意图识别专家。

最匹配的一项，并提取下游回答和查询需要的业务实体。

# Task

1. 上下文理解
   - 同时阅读会话历史和当前用户输入。
   - 对“它呢”“今天的呢”“按实际版呢”等省略表达，必须结合上一轮补全语义。
   - 不要在输出中改写用户问题，也不要虚构历史中不存在的信息。

2. 业务相关性判断
   - 判断问题是否属于企业铁路运输、煤炭运输、煤炭购销、
     企业制度规程、专业知识或经营数据范围。
   - 公众客运、火车票、个人出行、天气、娱乐、股票等通用问题，
     默认不属于本系统业务范围。
   - 不得仅依据“铁路”“煤炭”“合同”等单个关键词判断相关性。

3. 路由与意图识别
   - 首先从 BUSINESS、NON_BUSINESS、CLARIFY 中选择且只能选择一个 route。
   - route 为 BUSINESS 时，必须从 6 类业务意图中选择且只能选择一个 intent。
   - route 为 NON_BUSINESS 时，intent 必须为 null。
   - route 为 CLARIFY 时，必须生成一个明确的 clarification_question；若候选业务意图已经明确，可以保留 intent，否则填写 null。
   - 严禁创造新 route 或 intent 标签。

4. 实体提取
   - 只提取用户当前输入或会话历史中明确出现的信息。
   - 未提及的信息填写 null，不允许猜测。

5. 格式化输出
   - 只输出一个合法 JSON 对象。
   - 不要输出 Markdown 代码块或 JSON 以外的解释。
   - reasoning 只写简短的分类依据，不输出详细思维过程。
   - 无论 route 为何值，都必须输出完整结构。

# Route Taxonomy

请严格使用以下 3 个 route：

1. BUSINESS
   - 用户诉求属于企业铁路运输、煤炭运输、煤炭购销、制度规程、专业知识或经营数据范围；
   - 当前信息足以进入对应业务处理路径。

2. NON_BUSINESS
   - 用户诉求不属于本系统企业业务范围；
   - intent 必须为 null。

3. CLARIFY
   - 当前输入和会话历史不足以确认用户诉求，或者缺少执行当前业务请求必需的关键信息；
   - 必须生成一个简洁、可直接回答的 clarification_question；
   - 不得因为缺少非必要实体而过度澄清。

# Business Intent Taxonomy

route 为 BUSINESS 时，请严格使用以下 6 个业务意图：

1. POLICY_RULE_QA（政策制度与规程）

   适用范围：
   - 国家和行业政策法规；
   - 企业制度、管理办法和合同制度；
   - 铁路调度规程、行业规章和作业要求；
   - 询问“有什么规定”“是否允许”“依据哪一条”“应当如何执行”。

2. TRANSPORT_OPERATION_QA（运输生产与调度）

   适用范围：
   - 运行计划编制和调整；
   - 装车、编组、调运、发运、到达和卸车流程；
   - 货单、运单、货票的业务流程和使用方法；
   - 运输组织、车流组织和生产协调知识。

3. COAL_SALES_QA（煤炭购销与销售）

   适用范围：
   - 煤炭采购和销售业务；
   - 客户、供应商和合同管理；
   - 价格政策、合同条款、数质量验收、结算和扣罚；
   - 煤种、热值、硫分等指标在购销业务中的应用。

4. PROFESSIONAL_KNOWLEDGE_QA（专业知识与技术指导）

   适用范围：
   - 铁路、煤炭、港口、航运、电力和化工专业知识；
   - 专业概念、原理、指标含义和计算方法；
   - 不直接依赖某一份制度条文的技术解释。

5. BUSINESS_DATA_QUERY（经营数据查询与分析）

   适用范围：
   - 查询具体运行计划、列车、编组、合同、货单、运单和货票；
   - 查询运量、装车数、库存、完成率和到站状态；
   - 按时间、车站、客户、煤种或数据版本进行统计；
   - 同比、环比、历史、实际和模拟数据对比。

6. OTHER_BUSINESS（其他业务）

   适用范围：
   - 可以确认属于企业业务范围；
   - 当前不需要澄清，但无法稳定归入以上五类。

# Disambiguation Guidelines

1. 政策制度与规程 vs 专业知识与技术指导
   - “规程要求怎么做”属于政策制度与规程。
   - “这个概念是什么意思、为什么这样计算”属于专业知识与技术指导。

2. 运输生产与调度 vs 经营数据查询与分析
   - “运行计划怎么编制”属于运输生产与调度。
   - “查询今天的运行计划”属于经营数据查询与分析。

3. 运输生产与调度 vs 煤炭购销与销售
   - 关注装车、编组、调运和到站，属于运输生产与调度。
   - 关注客户、合同、价格、结算和数质量扣罚，属于煤炭购销与销售。

4. 企业货运 vs 公众客运
   - 企业煤炭货运和生产经营问题为业务相关。
   - 高铁票、旅客退票、个人旅行路线等公众服务问题为业务不相关。

5. 业务知识 vs 具体数据
   - 询问“是什么、怎么做、有什么规则”通常是知识问答。
   - 询问“多少、是否完成、当前到哪、某编号是什么状态”通常是数据查询。

6. 多轮省略
   - 如果当前问题本身不完整，但会话历史能够唯一补全含义，
     按补全后的业务诉求分类。
   - 不得因为当前一句缺少业务关键词而直接判为非业务。
~~~

## 7. 新旧 Prompt 对比

| 对比维度 | 初版 Prompt | 优化版 Prompt |
|---|---|---|
| 角色定位 | 通用企业知识库助手 | 铁路运输与煤炭经营意图识别专家 |
| 分类依据 | 铁路、煤炭、销售等领域关键词 | 下游回答 Prompt 和用户真实诉求 |
| 控制决策 | related 二值硬门控 | BUSINESS / NON_BUSINESS / CLARIFY 三态 route |
| 多轮理解 | 没有明确要求 | 必须结合会话历史补全省略表达 |
| 易混淆辨析 | 只列分类范围 | 明确六组业务边界 |
| 单据表达 | 统一 order_id | document_type + document_no |
| 时间与版本 | 一个笼统 time | time_range + data_version |
| 输出结构 | 非业务和业务结构不一致 | 所有 route 使用同一 JSON Schema |
| 标签约束 | 没有限制新增标签 | 只能使用 3 个 route 和 6 个业务 intent |
| 判定依据 | 无 reasoning | 输出简短、可审计的分类依据 |
| Few-shot | 无输入输出配对样例 | 覆盖知识、数据、非业务和多轮样例 |
| 下游用途 | 只能粗略分类 | 可直接选择专业 Prompt 并准备查询参数 |

## 8. 对比测评

### 8.1 测评口径

当前阶段采用样例级对照，不虚构尚未执行的整体准确率。

对比方法保持与 know-engine 一致：使用同一份标注数据集分别调用初版 Prompt 和优化版 Prompt，再比较 route、intent、entities、clarification_question 和输出合法性。进入实现阶段后，可以将数据集保存为独立 JSON 文件，并通过参数化测试重复执行，避免只挑选对新版有利的案例。

测评集应覆盖以下维度：

| 测试类型 | 关注问题 | 代表样例 |
|---|---|---|
| 基础意图 | 各意图能否稳定识别 | 调度规程、运行计划、煤炭合同 |
| 易混淆边界 | 相邻意图能否正确辨析 | 规则 vs 原理、流程 vs 数据 |
| 非业务边界 | 是否因关键词误判 | 高铁票、股票中的煤炭板块 |
| 单据实体 | 是否区分单据类型和编号 | 货单、运单、货票 |
| 时间口径 | 是否提取统计周期 | 今天、昨天、本月、去年同期 |
| 数据版本 | 是否识别历史、实际、模拟 | 按实际版、看模拟数据 |
| 多轮省略 | 是否结合历史理解短问题 | 它呢、今天的呢、按实际版呢 |
| 模糊输入 | 是否保持稳定兜底 | 我的运单、有问题、这个计划 |
| 输出协议 | 是否始终生成合法结构 | 非业务、实体缺失、长问题 |

后续真实运行时至少应记录：

- route 判断准确率；
- intent 分类准确率；
- 各意图召回率和混淆情况；
- 实体精确匹配率和关键实体召回率；
- JSON Schema 合法率；
- 非法标签生成次数；
- 多轮上下文一致率；
- 单次识别延迟和 Token 消耗。

| 今天神东站装了几列 | 可能只识别为铁路运输 | 经营数据查询与分析 | 识别隐含数据请求 |
| 重车周转时间是什么意思 | 可能识别为数据查询 | 专业知识与技术指导 | 区分指标概念和指标数值 |
| 按实际版呢 | 脱离上下文后归为其他 | route=BUSINESS，intent=BUSINESS_DATA_QUERY | 使用会话历史和数据版本 |
| 查一下我的运单 | 直接进入查询后因缺少编号失败 | route=CLARIFY，并询问运单号 | 用三态路由表达信息不足 |

从样例级对照可以确认，优化方向不是简单增加关键词，而是修正了意图分类的判断维度：

~~~text
初版：问题提到了哪个业务名词？

优化版：用户需要哪一种专业回答方式？
~~~

## 9. 为什么没有直接采用更细粒度的意图

讨论中还考虑过进一步拆分：

- 运行计划查询；
- 运输单据查询；
- 业务分析；
- 异常和应急处理。

这些分类在未来可能有价值，但当前没有全部进入 V1，原因是意图的存在必须有明确下游消费者。

如果两个意图最终：

- 使用相同回答 Prompt；
- 调用相同检索能力；
- 使用相同结构化查询入口；
- 遵守相同权限和数据口径；

那么提前拆分只会增加分类歧义和测试成本。

因此，当前继续采用 know-engine 风格的扁平业务意图和 Prompt 映射，同时用 route 替代 related 硬门控；等运行计划、运输单据、业务分析或异常处理拥有独立工具和回答策略后，再从现有类别中拆出。

## 10. 阶段性结论

这次优化带来的主要变化包括：

1. 将 related 二值硬门控升级为 BUSINESS / NON_BUSINESS / CLARIFY 三态 route；
2. 将意图分类从产业领域标签改为用户诉求和回答策略；
3. 明确企业煤炭货运与公众铁路服务的业务边界；
4. 区分政策规程、业务流程、专业知识和具体数据查询；
5. 使用真实铁路业务对象表达运行计划、编组、货单、运单和货票；
6. 增加时间范围、数据版本和经营指标等关键实体；
7. 要求意图识别结合会话历史理解省略和指代；
8. 增加易混淆辨析规则和正反 Few-shot；
9. 统一业务、非业务和澄清问题的结构化输出；
10. 保留简短 reasoning，便于定位误判，但不输出冗长思维链；
11. 以测试集和 bad case 驱动 Prompt 演进，而不是不断堆叠领域关键词。

当前的核心经验可以概括为：

> 企业意图识别的关键不是列出多少业务名词，而是先明确每个分类将触发什么 Prompt、检索策略或工具，再用真实业务边界和负例约束模型的选择。

## 11. 当前事实边界

本文记录的是 ke-engine 场景下的 Prompt 设计、代表性 bad case 和离线测评方案。

当前可以确认的是：

- know-engine 的扁平业务意图、实体提取和 Prompt 映射方式可以迁移；
- ke-engine 不复制 related 硬门控，使用 route 表达图控制决策；
- 原有汽车领域标签不能直接用于铁路运输和煤炭经营场景；
- 知识库目录不能直接等同于对话意图；
- 运行计划、编组、货单、运单、货票、时间范围和数据版本是必须进入样例设计的真实业务概念；
- 上述 bad case 能够作为后续新旧 Prompt 对照测试的回归样例。

当前尚未宣称的是：

- 优化版 Prompt 已经达到某个整体准确率；
- 所有实体字段均已进入生产代码；
- 细粒度业务意图已经全部实现；
- BUSINESS 分支的 RAG 和结构化查询已经完成。

待实际运行离线测试后，应将真实指标、失败样例和模型配置补充到本文，再形成最终测评结论。
